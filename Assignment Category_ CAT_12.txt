Project Theme
StudyNook is a full‑stack web application where students and library users can list study rooms they control (e.g., private rooms in a university library), and any registered user can browse, search, filter, and book those rooms for a specific date and time slot. The platform automatically prevents double‑booking using time‑conflict detection, allows room owners to manage their listings, and gives every user a dashboard to handle their own bookings. Secure JWT authentication is stored in HTTP‑only cookies, and the whole site is responsive and recruiter‑friendly.

4. CRUD Operations – Study Rooms
All CRUD operations manipulate the rooms collection. Any authenticated user can create a room; they become the room’s owner. Users can only edit or delete their own rooms (checked on the server by comparing req.user.id with the room’s owner field).
4.1 Add Room (Private Route /add-room)
* Form fields:
   * Room Name (text, required)
   * Description (textarea, required)
   * Image (image URLfrom internet)
   * Floor (text/number, e.g., “3rd Floor”)
   * Capacity (number, e.g., 4)
   * Hourly Rate (number, e.g., 5 – in dollars)
   * Amenities – a set of checkboxes. Options: Whiteboard, Projector, Wi‑Fi, Power Outlets, Quiet Zone, Air Conditioning. Selected values are joined into an array of strings.
* On success: toast “Room added successfully” and redirect to My Listings or Rooms page.
4.2 All Rooms Page (/rooms)
* Public route. Fetches all rooms from the DB.
* Display in the same card grid as the home page (keeping uniform image size).
* If no rooms match, display a friendly “No rooms found” message.
4.3 Room Details Page (/rooms/:id)
* Full details displayed:
   * Booking Count – a number that increments every time the room is booked (see challenge section)
   * “Book Now” button:
      * If user is logged in → opens the booking form (modal or inline).
      * If not logged in → button reads “Login to Book” and redirects to Login.
* Edit & Delete controls (only visible if current user is the room owner):
   * Edit button opens a pre‑filled form (modal or separate page) where all fields can be updated.
   * Delete button triggers a confirmation modal; after confirmation the room is permanently deleted.
4.4 Update Room (Owner only)
* API: PUT /api/rooms/:id (or PATCH)
   * Verifies owner matches req.user.id.
   * Accepts the updated fields (partial or full). Image URL can be changed by uploading a new picture (optional; if not changed, keep the existing URL).
* Success toast: “Room updated successfully”.
4.5 Delete Room (Owner only)
* API: DELETE /api/rooms/:id
   * Server checks ownership.
   * Before deleting, optionally use $pull to remove the room’s ID from any user’s bookings array (if any bookings reference this room).
   * Delete the room document.
* Success toast: “Room deleted successfully”.

5. Booking System
All bookings are stored in a bookings collection.
5.1 Book a Room (Private)
* Triggered by “Book Now” on the room details page.
* Booking form (modal or inline) contains:
   * Date – date picker; must be today or a future date.
   * Start Time – dropdown with hourly slots: 08:00, 09:00, …, 20:00.
   * End Time – dropdown, only times after the selected start time (same hour granularity). Minimum booking is 1 hour.
   * Total Cost – automatically computed in real‑time: (endHour - startHour) * hourlyRate.
   * Special Note (optional text).
   * Confirm Booking button.
* Conflict check:
   * Check if the selected room and time slot already has a confirmed booking.
   * If conflict exists, return an error; otherwise, create the booking successfully.
* Frontend shows a success toast “Room booked successfully!” and closes the modal.
5.2 My Bookings Page (Private /my-bookings)
* Fetch bookings where userId matches the logged‑in user, populated with room data (name, image).
* Display in a table or card list.:
   * Status badge: “confirmed” (green) or “cancelled” (red)
   * Cancel button – visible only if status is “confirmed” and the booking date is in the future (today’s future or later).
* Empty state: “You have no bookings yet.”
5.3 Cancel Booking (Private)
* Clicking “Cancel” opens a confirmation modal.
* On confirm, send PATCH /api/bookings/:id/cancel.
   * Server verifies the booking belongs to the user.
   * Updates status to "cancelled".
   * Uses $pull to remove the booking ID from the user’s bookings array.
   * (Optional) Could also $inc the room's bookingCount by -1 to keep it accurate, but not mandatory.
* Success toast: “Booking cancelled”.

7. Challenge Requirements
7.1 JWT with HTTP-Only Cookies
* On successful login (email/password or Google), generate a JWT containing { userId: user._id } (optionally role).
* Store the token in an HTTP-only cookie (httpOnly: true, secure: true in production, sameSite: 'strict').
* Create authMiddleware to:
   * Read token from req.cookies.token
   * Verify JWT
   * Attach req.user = { id: userId }
   * Return 401 Unauthorized if invalid or expired
* Apply middleware to all private routes (Add Room, Book Room, My Bookings, Edit/Delete, Cancel).
* Logout clears cookie using res.clearCookie('token').
